# 파트 1. 부정 효과 아이템 — Pawn에 효과를 "위임"하기

## 1-1. 아이템은 효과를 직접 적용하지 않는다

`ASlowItem`/`AReverseItem`은 `ABaseItem`을 상속하고 `ActivateItem`만 오버라이드한다. 중요한 건 **아이템이 `MoveSpeed`를 직접 건드리지 않는다**는 점이다. 대신 `AMyPawn`에 정의된 효과 메서드를 호출한다.

```cpp
// SlowItem.cpp
void ASlowItem::ActivateItem(AActor* Activator)
{
    if (Activator && Activator->ActorHasTag("Player"))
    {
        GEngine->AddOnScreenDebugMessage(-1, 2.0f, FColor::Blue, TEXT("Player Slowed"));
        if (AMyPawn* PlayerCharacter = Cast<AMyPawn>(Activator))
        {
            PlayerCharacter->GetSlow();      // ★ 효과 적용은 Pawn에게 위임
        }
    }
    DestroyItem();
}
```

```cpp
// ReverseItem.cpp — 구조가 동일, 호출 메서드만 다름
PlayerCharacter->ReverseInput();
```

### 💡 배운 점
- **`ActorHasTag("Player")` 가드 + `Cast<AMyPawn>` 이중 체크**: 태그로 "플레이어인가?"를 먼저 거르고, 캐스팅으로 "실제 타입이 맞는가?"를 확인. 둘 중 하나라도 실패하면 효과를 적용하지 않는다.
- 효과 상태(이동속도, 입력 배수)와 타이머는 **Pawn이 소유**해야 한다. 아이템은 곧바로 `DestroyItem()`으로 사라지므로, 아이템에 타이머를 두면 5초가 지나기 전에 파괴돼 효과 해제가 안 된다. → **수명이 짧은 객체에 지속 효과의 타이머를 두면 안 된다.**

## 1-2. Pawn 쪽 효과 로직 — 타이머로 켜고/끄기

`AMyPawn`은 `FTimerHandle` 두 개(`SlowTimer`, `ReverseInputTimer`)로 디버프의 지속/해제를 관리한다.

```cpp
void AMyPawn::GetSlow()
{
    if (!SlowTimer.IsValid())   // ★ 이미 슬로우 중이면 속도를 중복으로 깎지 않음
    {
        MoveSpeed *= 0.5;
    }
    GetWorldTimerManager().SetTimer(SlowTimer, this, &AMyPawn::EndSlowTimer, 5.0f, false);
}

void AMyPawn::EndSlowTimer()
{
    MoveSpeed *= 2;             // 원복
    GetWorldTimerManager().ClearTimer(SlowTimer);
}
```

```cpp
void AMyPawn::ReverseInput()
{
    if (!ReverseInputTimer.IsValid())
    {
        InputMult = -1;         // 입력 배수를 -1로 → 방향 반전
    }
    GetWorldTimerManager().SetTimer(ReverseInputTimer, this, &AMyPawn::EndReverseInput, 5.0f, false);
}

void AMyPawn::EndReverseInput()
{
    InputMult = 1;
    GetWorldTimerManager().ClearTimer(ReverseInputTimer);
}
```

`InputMult`는 이동 입력 처리에 곱해진다:

```cpp
void AMyPawn::Move(const FInputActionValue& Value)
{
    MoveInput = Value.Get<FVector2D>() * InputMult;   // -1이면 반대로
}
```

### 💡 배운 점
- **`Timer.IsValid()` 가드의 핵심 의도**: 슬로우 상태에서 슬로우 아이템을 또 먹어도 `MoveSpeed`가 0.25배로 곱쳐지지 않게 한다. 효과 자체는 중복 적용하지 않되, `SetTimer`는 매번 다시 호출되므로 **지속 시간만 5초로 갱신(리프레시)** 된다.
- 슬로우를 "0.5 곱하고 → 2 곱해서 원복"하는 방식은 **곱셈/역연산 쌍**이라 기준값(`MoveSpeed`)을 따로 저장하지 않아도 된다. 다만 중간에 다른 코드가 `MoveSpeed`를 바꾸면 원복이 어긋날 수 있다는 한계는 인지해 둘 것.

# 파트 2. 3D 오버헤드 위젯 — 상태를 머리 위에 띄우기

## 2-1. Pawn에 위젯 컴포넌트 3개

`AMyPawn`은 HP/Slow/Reverse 상태를 보여줄 `UWidgetComponent` 세 개를 가진다. 모두 **Screen Space**로 설정해 카메라를 향하도록 했다.

```cpp
OverheadWidget  = CreateDefaultSubobject<UWidgetComponent>(TEXT("OverheadWidget"));
OverheadSlow    = CreateDefaultSubobject<UWidgetComponent>(TEXT("OverheadSlow"));
OverheadReverse = CreateDefaultSubobject<UWidgetComponent>(TEXT("OverheadReverse"));

OverheadSlow->SetupAttachment(SkeletalMeshComp);
OverheadSlow->SetWidgetSpace(EWidgetSpace::Screen);   // 항상 카메라를 향함
```

## 2-2. 위젯이 읽어갈 "남은 시간" 게터를 BlueprintCallable로 노출

상태 전달의 핵심은 **C++에서 위젯을 직접 갱신하지 않고**, 위젯(블루프린트)이 매 프레임 Pawn의 게터를 폴링해서 애니메이션을 구동하도록 한 점이다.

```cpp
UFUNCTION(BlueprintCallable, Category = "Timer")
float GetSlowRemaining();

UFUNCTION(BlueprintCallable, Category = "Timer")
float GetReverseRemaining();
```

```cpp
float AMyPawn::GetSlowRemaining()
{
    return GetWorldTimerManager().GetTimerRemaining(SlowTimer);
}
```

> `GetTimerRemaining`은 타이머가 유효하지 않으면 **-1**을 반환한다. 위젯 쪽에서 이 값으로 "효과 진행 중인지" 여부와 게이지 진행률(`Remaining / 5.0`)을 동시에 계산할 수 있다.

## 2-3. HP는 C++에서 직접 텍스트 갱신 (대조 사례)

반대로 HP 텍스트는 값이 바뀌는 시점이 명확(데미지/회복)하므로 **이벤트 기반으로 C++가 직접 푸시**한다.

```cpp
void AMyPawn::UpdateOverheadHP()
{
    UUserWidget* Inst = OverheadWidget->GetUserWidgetObject();
    if (!Inst) return;
    if (UTextBlock* HPText = Cast<UTextBlock>(Inst->GetWidgetFromName(TEXT("OverHeadHP"))))
    {
        HPText->SetText(FText::FromString(FString::Printf(TEXT("%.0f / %.0f"), Health, MaxHealth)));
    }
}
```

### 💡 배운 점 — 두 가지 갱신 패턴의 선택 기준
| 상태 | 갱신 방식 | 이유 |
|------|-----------|------|
| HP | **이벤트 푸시** (`AddHealth`/`TakeDamage` 시 호출) | 변화 시점이 명확하고 띄엄띄엄함 |
| Slow/Reverse 남은 시간 | **위젯 폴링** (`GetXXXRemaining`) | 매 프레임 부드럽게 줄어드는 게이지 애니메이션이 필요 |

- `GetWidgetFromName(TEXT("..."))`로 위젯 블루프린트 내부 요소를 이름으로 찾는다 → **C++ 변수명과 위젯 BP의 위젯 이름이 정확히 일치**해야 한다. (`OverHeadHP` 같은 이름은 오타에 취약하니 주의)
- 메뉴 위젯에서도 같은 패턴이 보인다: `MainMenuWidgetInstance->FindFunction(FName("PlayGameOverAnim"))` + `ProcessEvent`로 **C++에서 블루프린트 애니메이션을 트리거** (`MyPlayerController.cpp`). → 애니메이션 재생을 C++가 명령할 수도 있다는 또 다른 연계 방식.

# 파트 3. 레벨 분리 & 단계별 스케일링

## 3-1. 레벨 = 맵, 진행 단계 = GameInstance에 보존

레벨을 코드 분기가 아니라 **물리적으로 다른 맵 파일**로 분리했다. 어떤 맵으로 갈지는 `AMyGameState::LevelMapNames` 배열이 들고 있다.

```cpp
// MyGameState.h
UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Level")
TArray<FName> LevelMapNames;     // 레벨별 맵 이름 목록 (에디터에서 지정)

int32 CurrentLevelIndex;
int32 MaxLevels;                 // = 3
```

레벨 종료 시 인덱스를 올리고, **다음 맵을 연다.** 맵을 열면 현재 월드의 액터(GameState 포함)는 전부 파괴되므로, **진행 단계는 `GameInstance`에 백업**해 둔다.

```cpp
void AMyGameState::EndLevel()
{
    if (UMyGameInstance* GI = Cast<UMyGameInstance>(GetGameInstance()))
    {
        AddScore(Score);
        CurrentLevelIndex++;
        GI->CurrentLevelIndex = CurrentLevelIndex;   // ★ 맵 전환에도 살아남도록 보존
    }
    GetWorldTimerManager().ClearTimer(LevelTimerHandle);

    if (CurrentLevelIndex >= MaxLevels)              // 3레벨 다 돌면 종료
    {
        OnGameOver();
        return;
    }
    if (LevelMapNames.IsValidIndex(CurrentLevelIndex))
    {
        UGameplayStatics::OpenLevel(GetWorld(), LevelMapNames[CurrentLevelIndex]);
    }
    else
    {
        OnGameOver();
    }
}
```

새 맵에서 `BeginPlay → StartLevel`이 돌 때, **GameInstance에서 단계를 복원**한다.

```cpp
void AMyGameState::StartLevel()
{
    // ...
    if (UMyGameInstance* GI = Cast<UMyGameInstance>(GetGameInstance()))
    {
        CurrentLevelIndex = GI->CurrentLevelIndex;   // ★ 보존해 둔 단계 복원
    }
    // 이 단계 값을 기준으로 스폰/난이도를 조정
}
```

## 3-2. 단계별 스케일링이 작동하는 원리

`CurrentLevelIndex`가 맵 전환을 넘어 연속적으로 유지되므로, 각 레벨 맵은 이 인덱스를 기준으로 **난이도를 단계별로 다르게** 구성할 수 있다:

- **맵 자체의 차이**: 레벨별로 다른 맵을 쓰므로 `SpawnVolume` 배치, 데이터테이블(`ItemDataTable`)의 아이템 확률(부정 효과 아이템 비중 등)을 맵마다 다르게 세팅 → 후반 맵일수록 Slow/Reverse 같은 디버프 아이템 비중을 높이는 식의 스케일링이 가능.
- **공통 루프 규칙**: 레벨당 제한 시간(`LevelDuration = 30s`)과 스폰 수(`ItemToSpawn = 40`) 기반으로 진행하고, 코인을 다 모으면 즉시 다음 레벨로.

```cpp
// SpawnVolume — 데이터테이블 확률 기반 스폰. 맵마다 다른 테이블을 꽂으면 난이도가 달라짐
FItemSpawnRow* ASpawnVolume::GetRandomItem() const
{
    // 각 Row의 SpawnChance를 누적해 가중치 랜덤 추첨
    const float RandValue = FMath::FRandRange(0.0f, TotalChance);
    // ...
}
```

### 💡 배운 점
- **맵 전환 = 월드 전체 리셋**. `OpenLevel` 한 줄에 모든 액터가 파괴되므로, "다음 단계로 넘겨야 하는 값"은 반드시 `GameInstance`(게임 종료까지 살아있는 단 하나의 객체)에 둬야 한다. → 어제 정리한 [[2026-06-15-게임루프]]의 "맵이 바뀌어도 유지돼야 하는 데이터는 GameInstance" 원칙의 실전 적용.
- `LevelMapNames.IsValidIndex(...)` 체크로 **배열 범위를 벗어나면 게임오버**로 안전하게 빠진다 → 맵을 빠뜨려 설정해도 크래시 대신 정상 종료.
- `MaxLevels`(3)와 `LevelMapNames`(맵 개수)가 **이중으로 종료 조건을 결정**한다. 둘이 어긋나면(`MaxLevels`는 3인데 맵은 2개) 의도와 다르게 끝날 수 있으니, 한쪽을 단일 진실 원천으로 삼는 리팩터링을 고려할 것.

# 4. 오늘의 회고

## 잘한 점
- **책임 분리**: 아이템(트리거) / Pawn(상태·타이머 소유) / 위젯(읽기 표시)로 3분할 → 새 디버프 아이템을 추가할 때 Pawn에 메서드 하나만 더하면 됨.
- **상태 전달 패턴 두 가지를 상황에 맞게 사용**: 변화가 띄엄띄엄한 HP는 이벤트 푸시, 매 프레임 줄어드는 게이지는 위젯 폴링.

## 개선/주의할 점
- 슬로우 효과를 `MoveSpeed *= 0.5 / *= 2`로 처리 → 기준값을 별도 보관하지 않아 다른 시스템이 `MoveSpeed`를 바꾸면 원복이 어긋날 위험. **`BaseMoveSpeed`를 따로 두고 `MoveSpeed = BaseMoveSpeed * SlowMult` 식으로 재계산**하는 방식이 더 견고.
- `SlowItem`에 `SlowMult` 프로퍼티가 선언돼 있지만 실제 효과(`GetSlow`)는 하드코딩된 0.5를 사용 → 프로퍼티가 효과에 반영되도록 `GetSlow(float Mult)`로 인자를 넘기면 에디터에서 강도 조절 가능.
- `MaxLevels`와 `LevelMapNames` 길이가 별도로 관리됨 → 단일 진실 원천으로 통합 고려.
